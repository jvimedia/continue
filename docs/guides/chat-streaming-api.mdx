## What This Is

The Continue VS Code extension can expose the chat session running in its sidebar over a small local API. External clients can:

- Watch the conversation live, token-by-token, as the assistant responds (Server-Sent Events or WebSocket)
- Read a snapshot of the current session's history
- Send a new message into the chat, exactly as if it were typed into the sidebar

This is what powers the reference [iOS app](https://github.com/continuedev/continue/tree/main/apps/ios/ContinueChat) in this repo and the built-in Telegram bot relay, but the API itself is plain HTTP/SSE/WebSocket, so you can build your own client against it.

<Warning>
  This exposes the content of your chat (including code context) to anything
  that holds the API token. Keep the server bound to `127.0.0.1` unless you
  understand the risk of exposing it further, and never share the token.
</Warning>

## How It Works

The server doesn't reimplement chat logic - it mirrors exactly what the Continue sidebar already sends and receives internally:

- When you (or an external client) send a message, it goes through the same `userInput` mechanism Continue already uses for things like "Ask Continue about this terminal error." It shows up in the real sidebar, gets sent to your configured model, and gets saved to session history like any other message.
- Every chunk of the assistant's streamed reply is mirrored out to connected clients the moment it's sent to the sidebar webview.

There is one architectural consequence worth knowing: a message you type **directly in the VS Code sidebar** is only visible to the API once the *next* chat turn starts (Continue's webview holds the freshly-typed user turn locally until it issues the underlying chat request), whereas messages sent via `POST /message` or over the WebSocket appear immediately as a `user_message` event, and the response streams back over the same connection.

## Enabling the Server

The easiest way is the Continue sidebar: **Settings → Remote Access**. There you can toggle the server, change the port, allow LAN access, and copy the server URL and token.

Equivalently, in VS Code settings:

- `continue.chatApi.enabled`: `true`
- `continue.chatApi.port` (optional, default `65433`)
- `continue.chatApi.host` (optional, default `127.0.0.1`; the "Allow LAN Access" toggle sets this to `0.0.0.0`)
- `continue.chatApi.mdns` (optional, default `true`): advertise the server via Bonjour/mDNS

The token can also be retrieved with **Continue: Show Chat API Token** from the Command Palette. It is generated once per VS Code installation and stored in VS Code's encrypted `SecretStorage` - it is not written to your settings file.

If the configured port is already taken (typically by the Chat API server of another VS Code window), the server walks forward to the next free port - so every open window gets its own server, and clients can list and join any of them.

## Auto-Discovery (Bonjour/mDNS)

When the server is enabled, LAN access is on (host `0.0.0.0`), and `continue.chatApi.mdns` is `true`, the extension advertises the server on the local network as a `_continuejv._tcp` Bonjour service. Clients like the iOS app can then find your editor without you typing an IP address. The advertisement includes the workspace name in its TXT record, so a client can show *which project* each VS Code window has open. Loopback-bound servers are never advertised, since other devices couldn't reach them anyway.

## Remote GUI

Beyond the raw JSON API, the server hosts the extension's **actual sidebar GUI** at `GET /gui?token=<token>` - the same built React app the sidebar renders, with identical markdown formatting, mode selector (Chat/Plan/Agent), model picker, session tabs, and settings. Instead of VS Code's webview transport, the page bridges the same webview protocol over a WebSocket at `/gui-ws`; on the extension side those messages feed into the exact handler chain the sidebar uses, so the remote GUI is a full Continue client sharing the window's config, sessions, and tools.

- `GET /gui?token=<token>&sessionId=<id>` opens a specific session after loading.
- Imperative messages (e.g. the `userInput` injection used by `POST /message` and the Telegram relay) are delivered only to the real sidebar, never to remote GUI clients, so inputs are never processed twice.
- Each remote GUI is its own instance of the app: a turn you start on one client streams live on that client, while other clients see the updated session on their next session load rather than token-by-token.

Two supporting endpoints power client-side window/session pickers:

- `GET /info`: `{ workspaceName, workspacePaths, appName, extensionVersion, currentSessionId, port }` - identifies the VS Code window and which session its sidebar currently shows.
- `GET /sessions`: `{ sessions: [{ sessionId, title, dateCreated, workspaceDirectory }] }` - all sessions of this window, so a client can offer "join session".

## Telegram Bot Relay

The extension can also relay the chat session to a Telegram bot, with no separate server process:

1. Create a bot with [@BotFather](https://t.me/BotFather) and copy its token.
2. In the Continue sidebar under **Settings → Telegram Bot**, enable the relay and paste the bot token (stored in `SecretStorage`; also settable via **Continue: Set Telegram Bot Token**).
3. Message your bot once - it replies with your chat ID. Add that ID to **Allowed Chat IDs**.

From then on, messages you send to the bot from an allowed chat are injected into the sidebar conversation, and the assistant's replies (plus user messages from other clients) are sent back to every allowed chat. The relay uses Telegram long polling, so it works behind NAT and does not require the HTTP API server itself to be enabled.

<Warning>
  Anyone in an allowed Telegram chat can read and write your Continue chat
  session. Only add chat IDs you control.
</Warning>

## Authentication

Every request (except `GET /health`) must include the token, either as a bearer header or a query parameter:

```bash
curl -H "Authorization: Bearer <token>" http://127.0.0.1:65432/session
# or
curl "http://127.0.0.1:65432/session?token=<token>"
```

WebSocket clients pass it as a query parameter when opening the connection: `ws://127.0.0.1:65432/ws?token=<token>`.

## REST Endpoints

### `GET /health`

No auth required. Returns `{ "ok": true }` if the server is up.

### `GET /session`

Returns a snapshot of the currently active session in the sidebar.

```json
{
  "sessionId": "1f9e2b3a-...",
  "session": {
    "sessionId": "1f9e2b3a-...",
    "title": "Fix the flaky test",
    "history": [
      { "message": { "role": "user", "content": "..." }, "contextItems": [] },
      { "message": { "role": "assistant", "content": "..." }, "contextItems": [] }
    ]
  }
}
```

If no chat has been started yet in this VS Code window, `sessionId` and `session` are both `undefined`.

### `POST /message`

Sends a message into the chat, as if typed into the sidebar input box.

```bash
curl -X POST http://127.0.0.1:65432/message \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"input": "What does this repo do?"}'
```

Returns `{ "ok": true }` immediately - it does not wait for the assistant's reply. Subscribe to `/events` or `/ws` to see the response stream in.

## Streaming: `GET /events` (Server-Sent Events)

Opens a long-lived `text/event-stream` connection. Every event is a JSON-encoded `data:` line:

```
data: {"type":"user_message","turnId":"a1b2...","timestamp":1750000000000,"data":{"role":"user","content":"What does this repo do?"}}

data: {"type":"assistant_delta","turnId":"a1b2...","timestamp":1750000000010,"data":{"content":[{"role":"assistant","content":"This "}],"status":"success"}}

data: {"type":"assistant_delta","turnId":"a1b2...","timestamp":1750000000020,"data":{"content":[{"role":"assistant","content":"repo is..."}],"status":"success"}}

data: {"type":"assistant_done","turnId":"a1b2...","timestamp":1750000001000,"data":{"content":[{"role":"assistant","content":"This repo is..."}],"status":"success"}}
```

### Event Types

| `type` | Meaning |
| --- | --- |
| `user_message` | A new user turn was sent (from any client, including the sidebar itself). `data` is the `ChatMessage`. |
| `assistant_delta` | An incremental chunk of the assistant's reply. `data.content` is the latest chunk (a `ChatMessage` or array of them, matching Continue's internal streaming format). |
| `assistant_done` | The turn finished. `data.status` is `"success"` or `"error"`. |
| `session` | Session metadata changed (e.g. title, new session created). |

`turnId` is stable across all events belonging to the same chat turn - use it to group deltas into a single message on the client side.

### JavaScript example

```js
const es = new EventSource("http://127.0.0.1:65432/events?token=" + token);
const buffers = new Map();

es.onmessage = (e) => {
  const event = JSON.parse(e.data);
  if (event.type === "assistant_delta" || event.type === "assistant_done") {
    const chunks = [].concat(event.data.content ?? []);
    const text = chunks.map((c) => c.content).join("");
    buffers.set(event.turnId, (buffers.get(event.turnId) ?? "") + text);
    render(event.turnId, buffers.get(event.turnId));
  }
};
```

## Streaming: `GET /ws` (WebSocket)

Same events as SSE, pushed as WebSocket text frames. Additionally accepts inbound messages to send chat input without a separate HTTP request:

```json
{ "type": "message", "input": "What does this repo do?" }
```

```js
const ws = new WebSocket(`ws://127.0.0.1:65432/ws?token=${token}`);
ws.onmessage = (e) => console.log(JSON.parse(e.data));
ws.onopen = () => ws.send(JSON.stringify({ type: "message", input: "Hello" }));
```

## Reference Clients

- **iOS app** (`apps/ios/ContinueChat`): discovers open editors via Bonjour (showing each window's project) or takes an explicit host/URL and port, lists that window's sessions so you can pick one to join, and then renders the [Remote GUI](#remote-gui) - the identical sidebar interface - in a native shell. See its README for build instructions (XcodeGen).
- **Telegram bot**: built into the extension itself - see [Telegram Bot Relay](#telegram-bot-relay) above.

## Troubleshooting

- **`ECONNREFUSED`**: the server isn't enabled, or VS Code hasn't finished activating the extension yet. Check `continue.chatApi.enabled` and the **Continue Chat API** output channel (View → Output).
- **`401 Invalid or missing API token`**: re-run **Continue: Show Chat API Token** and make sure you're using the latest value - it's regenerated if VS Code's secret storage is cleared.
- **No `user_message` event for something you typed in the sidebar**: expected - the sidebar only reveals a turn to the API once it sends the underlying chat request. Use `POST /message` for anything that should show up immediately.
- **Connecting from a physical iOS device or a separate machine (e.g. for the Telegram bot)**: set `continue.chatApi.host` to `0.0.0.0` and reach your machine's LAN IP, or use a tool like an SSH tunnel/Tailscale rather than exposing the port publicly.
